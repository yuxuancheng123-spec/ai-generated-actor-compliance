# LINDDUN Threat Model for AI Actor Compliance Workflow

This threat model focuses on privacy risks in a synthetic media platform that accepts face, voice, video, motion, consent, and publication metadata. It complements the misuse cases in `privacy_engineering/threat_model_misuse_cases.md`.

| LINDDUN category | Affected data flow node | Affected party | Misuse scenario | Harm | Existing control | Residual risk | Proposed mitigation | Implementation status |
|---|---|---|---|---|---|---|---|---|
| Linkability | Intake frontend, case database, evidence records | Performer, ordinary person | Multiple uploads, consent records, and generated outputs are linked across campaigns, revealing reuse patterns | Identity profiling, purpose creep, loss of contextual privacy | Case IDs, evidence references, retention fields | Case metadata still links identity across review events | Purpose-bound media IDs, campaign-level access limits, retention enforcement | Partially implemented through case/evidence records and retention fields |
| Identifiability | Source media, biometric/likeness pipeline, consent vault | Performer, public figure, ordinary person | Face or voice samples identify a person even when the generated output is labeled synthetic | Unauthorized identity exposure, publicity-rights risk, personal safety risk | Represented-person fields, identity verification status, consent scope | The prototype does not store encrypted media or biometric templates | Encrypt media, minimize template retention, separate media store from review metadata | Documented; not implemented in local SQLite prototype |
| Non-repudiation | Audit log, report export, review endpoint | Requester, reviewer, platform | A requester denies uploading source media or a reviewer denies approving a risky case | Accountability gap, weak incident investigation | Audit log for case creation, assessment, review update, deletion request | Header-based role model is not strong identity proof | Signed audit events, reviewer identity binding, immutable log storage | Partially implemented through audit log; production identity not implemented |
| Detectability | API logs, review status, report export | Performer, requester | Existence of a sensitive likeness review becomes visible to internal users who do not need to know | Sensitive inference, reputational harm | Role concept, reviewer status | No fine-grained field-level access control | Need-to-know permissions, case sensitivity labels, audit review for accesses | Defined as role model; not production-authenticated |
| Disclosure of information | Evidence records, report, audit log | Performer, rights holder, requester | Consent documents or source media references leak through report exports or broad database access | Contract disclosure, personal data exposure, commercial confidentiality breach | Evidence references separated from rule trace, soft deletion | Evidence URIs could still expose sensitive locations | Redact exports, encrypt evidence store, signed URLs, data minimization | Partially implemented through references only |
| Unawareness | Intake frontend, disclosure plan, report | Viewer, represented person | Viewer cannot tell AI content is synthetic, or depicted person is unaware of digital replica use | Deception, loss of agency, reputational harm | Visible label, machine-readable metadata, watermark rules | Labels can be stripped or omitted in downstream reposts | Export-resistant watermarking, C2PA/content credentials, complaint portal | Partially implemented through rules and documentation |
| Non-compliance | Rules engine, consent record, distribution platform | Platform, performer, regulators | System approves a case despite missing consent, missing labels, or jurisdiction-specific requirements | Regulatory exposure, rights violation, takedown liability | Rule trace, hard stops, missing-information list, framework references | Rule coverage is incomplete and not jurisdiction-exhaustive | Legal review, rule version governance, policy test cases, change management | Implemented as transparent prototype rules |
| Linkability | External vendor, model training pipeline | Performer, ordinary person | Vendor links performer media to model-improvement datasets across customers | Unauthorized training, irreversible identity replication | Training-use authorization field and rule R-05 | Vendor behavior is outside local prototype boundary | Vendor audit rights, data-use ledger, contract controls, deletion attestations | Documented; rule hard stop implemented |
| Identifiability | Complaint intake, evidence preservation store | Victim, rights holder | A stolen-face complaint exposes victim identity to the requester or unrelated reviewers | Retaliation, privacy harm | Incident response playbook and audit log | Complaint workflow is not implemented in API yet | Dedicated complaint role, redaction, limited disclosure workflow | Documented; future API work |
| Non-compliance | Labeling and provenance service | Viewer, regulator, platform | Visible label is removed after download and repost, breaking transparency obligations | Viewer deception, regulatory breach | Watermark/provenance rule R-09 | Browser demo and backend do not detect reposts | Hash registry, platform scanning, C2PA verification, repost detection | Partially implemented through provenance fields |
| Non-repudiation | Consent vault | Performer, platform | A malicious requester uploads forged consent or claims verbal permission | False authorization, identity theft | Consent status enum, evidence verification status, reviewer path | Prototype does not verify documents | Signer verification, performer callback, document authenticity checks | Documented; not implemented |
| Disclosure of information | Insider access to case database | Performer, requester | Internal operator views face/voice case details without operational need | Privacy breach, insider misuse | Role definitions and audit events | No user-level access logging beyond role header | Just-in-time access, field-level permissions, anomaly detection | Documented; role model only |

## Priority Mitigations

1. Replace role headers with authenticated reviewer identity and scoped permissions.
2. Split sensitive media storage from case metadata and use encrypted evidence references.
3. Add report redaction modes for external export.
4. Add complaint and affected-person notification workflow.
5. Add rule-version governance and regression tests for policy updates.
